# Servo dynamics identification + feedforward streaming (layers 3/4)

## Context

The servo tuning toolchain has four layers:

1. **Drive parameter access** (SDO read/write from host, config-as-code param
   push) — in progress, separate effort.
2. **Telemetry capture** (per-DC-cycle channel recording + offline analysis) —
   in progress, separate effort.
3. **Dynamics identification** — this design.
4. **Feedforward streaming** (CiA402 velocity offset 60B1h + torque offset
   60B2h) — this design.

Architecture decision (agreed): **hybrid**. The host owns identification and
feedforward — the parts that need kinematics knowledge. The drive keeps the
current loop, adaptive notch filters, and disturbance observer, and runs
manually-set, telemetry-verified gains.

Why feedforward is the CoreXY answer: per-motor effective inertia on CoreXY
depends on what the other motor is doing. With `a = x + y`, `b = x − y`,
toolhead mass `m_t`, gantry mass `m_g`:

```
τ_a ∝ (m_t/2 + m_g/4)·ä − (m_g/4)·b̈
```

A pure X move (ä = b̈) leaves motor A feeling `m_t/2`; a pure Y move
(b̈ = −ä) leaves it feeling `m_t/2 + m_g/2`. No single drive-side inertia
ratio (C00.06) is correct for both, so the drive's built-in inertia auto-tune
and internally-derived feedforward are structurally wrong on CoreXY. Host-side
torque feedforward computed from both motors' accelerations is exactly right
for every move direction, and the drive's feedback loop is left handling only
disturbances — which direction-agnostic moderate gains do fine.

Target drive: STEPPERONLINE A6-EC (EtherCAT, CSP mode, CiA402). Bench today:
Neptune 3 Pro, one servo on X (Cartesian — scalar case). Next: Trident CoreXY
(coupled 2×2 case).

## The dynamics model — motor space, constant coefficients

All dynamics are expressed in **motor-stream coordinates** (the per-motor
trajectories the endpoint already receives — post-kinematics, in mm of motor
travel):

```
τ_i = Σ_j M_ij·α_j + b_i·ω_i + c_i·sgn(ω_i)
```

- `τ_i` — torque feedforward for motor i, in 60B2h native units (0.1% of
  rated torque).
- `α_j`, `ω_j` — acceleration (mm/s²) and velocity (mm/s) of motor j's
  stream, evaluated from the Bézier pieces.
- `M` — constant mass matrix, units (0.1% rated torque)/(mm/s²). Symmetric.
  Diagonal for Cartesian; nonzero off-diagonals encode CoreXY coupling.
- `b_i` — viscous friction, (0.1% rated)/(mm/s).
- `c_i` — Coulomb friction, 0.1% rated.

Key properties:

- **Both Cartesian and CoreXY have constant M** — no configuration
  dependence, so runtime evaluation is a dot product per motor per cycle.
- **The endpoint never learns the kinematics.** It receives numbers (M, b, c)
  at claim time, exactly like `counts_per_mm` today. "CoreXY-ness" is just
  which off-diagonal entries are nonzero; the endpoint does the weighted sum
  without interpreting it.
- **No physical constants in config.** Masses, pulley radius, rated torque
  never appear: identification regresses measured torque (6077h, 0.1%-rated
  units) against motor-stream kinematics (mm/s², mm/s), so the fitted
  coefficients come out directly in the units the runtime consumes.

## Layer 3 — identification

### Excitation

A generator script emits G-code move sequences run through the normal planner
under layer-2 capture: constant-acceleration triangle strokes at several
acceleration levels and speeds, per Cartesian axis, plus diagonals on coupled
kinematics. No dedicated excitation mode in the endpoint — the planner
produces exactly-known trajectories, respects machine limits, and exercises
the real streaming path. Inertia and friction are DC-to-low-frequency
parameters; PRBS/chirp excitation buys bandwidth we do not need.

### Fit

Linear least squares of measured per-motor torque against the model above.

- Regressors (α, ω) come from the **commanded** trajectory — exactly known,
  noise-free. 606Ch measured velocity is kept as an alternative regressor for
  cross-validation; large divergence between the two fits indicates tracking
  error is polluting the regression (lower gains or slow the excitation).
- The **structure** of M is supplied per kinematics: scalar for a single
  Cartesian servo; symmetric 2×2 with equal diagonals for CoreXY A/B. The
  fitter takes the structure as input and fits only the free coefficients.
- Asymmetric Coulomb friction (forward ≠ reverse) is fitted; the drive's own
  friction compensation (C02.6B/C02.6C) is the eventual consumer if we choose
  to delegate it, but v1 keeps friction in the host FF term.

### Outputs

1. **Dynamics profile** — TOML file, one per printer, version-controlled by
   the user alongside printer config:

   ```toml
   # generated by servo-ident; units: torque in 0.1% rated, motion in mm
   version = 1
   axes = ["x"]               # motor-stream names, order fixes matrix indices
   mass = [[0.0123]]          # M, (0.1% rated)/(mm/s²), row-major
   viscous = [0.0045]         # b per motor
   coulomb_fwd = [1.2]        # c per motor, forward
   coulomb_rev = [-1.1]       # c per motor, reverse
   fit_rms_residual = [0.8]   # 0.1% rated — fit quality, fail loudly if absurd
   ```

2. **Recommended C00.06** (drive load inertia ratio) — derived for the
   drive's own bookkeeping using motor datasheet rotor inertia + rated torque
   + rotation_distance (tool inputs, not runtime config):
   `J_total = M_ii · (T_rated/1000) · rot_dist/(2π)`, ratio =
   `(J_total − J_rotor)/J_rotor · 100`. On CoreXY the tool reports the
   light-direction and heavy-direction values; recommendation is the
   **light-direction** value (loop gain scales with C00.06 — the light case
   is the stability-critical one; FF carries the heavy case).

3. Fit report: residuals per stroke, torque saturation incidents, condition
   number of the regression.

### Validation

- **Synthetic first**: simulate captures from a known M/b/c plus noise and
  quantization; the fitter must recover the truth within tolerance. This is
  the unit-test substrate and does not wait on the bench.
- **Neptune cross-check**: the drive's built-in F30.10 inertia auto-tune is
  valid on a constant-inertia Cartesian axis — run both and compare the
  implied inertia ratio. Agreement validates the whole pipeline before it is
  used on the Trident where F30.10 has no ground truth to offer.

## Layer 4 — feedforward streaming

### PDO remap (fixed 1701h/1B01h → variable 1600h/1A00h)

The drive's fixed mappings cannot carry 60B1h at all. Variable mapping
(1600h RxPDO / 1A00h TxPDO, ≤10 objects, ≤40 bytes each) is configured by
SDO in PRE-OP — the same window where `ec_rt_bringup` already writes opmode
and sync params — and is not EEPROM-retained, which costs nothing since
bring-up rewrites comm SDOs on every claim.

New mappings (existing objects kept in current order, additions appended):

| RxPDO 1600h (18 B, 6 obj) | TxPDO 1A00h (32 B, 10 obj) |
|---|---|
| 6040h controlword u16 | 603Fh error code u16 |
| 607Ah target position i32 | 6041h statusword u16 |
| 60B8h touch probe fn u16 | 6064h position actual i32 |
| 60FEh:01 phys outputs u32 | 6077h torque actual i16 |
| **60B1h velocity offset i32** | 60F4h following error i32 |
| **60B2h torque offset i16** | 60B9h tp status u16 |
| | 60BAh tp1 pos i32 |
| | 60BCh tp2 pos i32 |
| | 60FDh digital inputs u32 |
| | **606Ch velocity actual i32** |

Sync-manager assignment: 1C12:01 = 1600h, 1C13:01 = 1A00h (clear, write
entries, write counts, reassign — per manual §8.3.1). `out_t`/`in_t` in
`bench/libecrt.c` grow accordingly. 6074h (drive-internal torque reference)
is the acknowledged sacrifice at the 10-object limit; revisit only if a
bench question demands it.

Drive-side FF routing joins the existing bring-up SDO writes (layer 1 not
required): C01.13 = 5 (speed FF source = communication), C01.16 = 5 (torque
FF source = communication), C01.14/C01.17 = 1000 (100.0% scale — verify on
bench that the percentage applies to communication sources), C01.15/C01.18
FF filters initially at default 318 Hz (each filter adds lag to an exact
signal; raising/disabling is a bench-tune item once telemetry can show the
difference).

New FFI surface: `ec_rt_set_velocity_offset(i32 /* counts/s */)`,
`ec_rt_set_torque_offset(i16 /* 0.1% rated */)`,
`ec_rt_get_velocity_actual()`, `ec_rt_get_torque_actual()`.

### Acceleration evaluation

`runtime::motion_core` gains second-derivative evaluation of the armed cubic
piece (same Horner path; acceleration of a cubic is piecewise-linear). The
walker returns position, velocity, acceleration per cycle. Position semantics
are unchanged — byte-identical stepper hot path is not at risk (motion_core
is shared; the addition is a new function, not a modification of existing
codegen — verify with the existing disasm check).

### Per-cycle computation (endpoint)

Each DC cycle, after walking all axis rings:

1. `vel_offset_counts = ω_i · counts_per_mm` → 60B1h.
2. `τ_i = Σ_j M_ij·α_j + b_i·ω_i + c_i·sgn(ω_i)` → clamp → 60B2h.

Clamp: configurable, default ±300 (±30.0% rated). Saturation increments a
counter exposed via StatusHeartbeat — visible, not silent, but not a fault:
saturation during aggressive tuning experiments is expected behavior, while a
*persistently* saturating profile is a miscalibration the telemetry makes
obvious. NaN/non-finite FF output is a hard fault (endpoint latches, drive
disabled) — that is a bug, not a tuning artifact.

### Dynamics profile delivery

Velocity FF is purely kinematic (ω comes from the curve, no fitted constants)
and is gated by its own spawn flag, `--velocity-ff` — this is what lets
rollout step 2 run before any identification exists. Torque FF is gated by
`--dynamics-profile <path>` (same delivery channel as `--counts-per-mm`).
The endpoint parses and validates the profile at startup: axis count must
match its ring count, M must be symmetric and positive-definite, all values
finite. Any violation fails the claim loudly. Neither flag ⇒ both offsets
written as 0 — bit-identical to today's drive behavior. Both flags are
derived by klippy from `[servo_*]` config keys, so enablement is printer
config, not deploy-script state.

### Coupled motors not co-located: shadow axes

Torque FF requires α_j for every motor coupled through M. Cases:

- Neptune (scalar M): no coupling — works with one ring.
- CoreXY, both servos on one bus: one endpoint process, two axis rings —
  both curves in hand. (Multi-slave support in `libecrt.c` — slave index is
  hard-coded to 1 today — is **deferred to the Trident bring-up design**;
  this design keeps the dynamics/FF code N-axis from day one so that step
  adds bus handling, not math.)
- CoreXY mixed (servo A + stepper B) or servos on separate buses: the host
  streams the coupled motor's curve to the endpoint as a **shadow axis** — an
  additional FF-only piece ring evaluated by the same walker but bound to no
  drive output. Same piece protocol, no format changes. Deferred with
  multi-slave support; named here so the profile format (axis list) and ring
  layout anticipate it.

### Rollout order

1. PDO remap + FFI, offsets hardwired to 0 — behavior-identical soak.
2. Velocity FF on Neptune (no dynamics profile needed — ω is kinematic).
   Wrong values degrade tracking, nothing else; following error before/after
   is the metric.
3. Identification run on Neptune; cross-check vs F30.10.
4. Torque FF on Neptune from the fitted profile; following-error comparison.
5. Trident CoreXY (own design: multi-slave + shadow axes), where the
   cross-term earns its keep.

## Interface contracts with layers 1–2 (parallel work)

1. **`bench/libecrt.c` ownership**: this design takes the PDO remap; layer-2
   telemetry builds on the remapped TxPDO and gains 606Ch velocity-actual for
   free. If the telemetry work needs to touch the mapping first, coordinate
   before either lands.
2. **Capture format**: the fitter core consumes time-aligned arrays of
   (t, target position, position actual 6064h, torque actual 6077h,
   velocity actual 606Ch). The adapter from the layer-2 capture format to
   these arrays is the only fitter component that waits on layer 2; it is
   kept as a thin boundary. Until it exists, the fitter runs on synthetic
   captures.
3. **Drive param push (layer 1)**: C01.13/14/16/17 land in `libecrt.c`
   bring-up for now; when layer 1's declarative param push exists, they
   migrate there (config-as-code) and the bring-up hardcoding is removed.

## Error handling

Fail loudly throughout, per project policy:

- Dynamics profile parse error, axis-count mismatch, asymmetric or
  non-positive-definite M, non-finite values ⇒ claim failure with a message
  naming the file and the violated invariant.
- Non-finite FF output at runtime ⇒ latched endpoint fault (existing fault
  path), drive disabled.
- Torque clamp saturation ⇒ counted and reported, not faulted (expected
  during tuning; the report is the loudness).
- Fitter: regression condition number above threshold, or residual RMS above
  a sanity bound ⇒ the tool refuses to emit a profile and says why
  (insufficient excitation, saturated torque, tracking divergence).

## Testing

- `motion_core` acceleration eval: unit tests against analytic derivatives of
  known cubics, including piece boundaries (acceleration is discontinuous
  across pieces — assert evaluation uses the armed piece only).
- FF computation: unit tests of the dot product, friction sign handling at
  ω = 0 (use a small dead band around zero rather than sgn-chatter), clamp
  and saturation counting, scalar and 2×2 profiles.
- Profile parsing/validation: reject each invariant violation; round-trip a
  known profile.
- Fitter: synthetic-capture recovery within tolerance under noise +
  quantization; structure enforcement (CoreXY fit returns equal diagonals);
  refusal paths (poor excitation ⇒ no profile).
- PDO remap: stub-level test that `out_t`/`in_t` layouts match the mapping
  tables byte-for-byte (compile-time asserts on struct sizes/offsets in C).
- Stepper-path safety: existing disasm check confirms the MCU stepper hot
  path is byte-identical after the motion_core addition.

## Explicitly out of scope

- Multi-slave EtherCAT bus support and shadow-axis streaming (Trident
  CoreXY bring-up design).
- Automated gain optimization (layer "C" — scripted sweeps scoring layer-2
  metrics). The hybrid architecture leaves room; nothing here forecloses it.
- Delegating friction to the drive's C02.6x compensation.
- Layers 1–2 themselves.
